## 🛠️ Usage Patterns

### Pattern 1: Install and Run (Fastest)

```bash
# Install from PyPI
pip install skillopt

# Configure API
export AZURE_OPENAI_ENDPOINT="https://your-resource.openai.azure.com/"
export AZURE_OPENAI_API_KEY="your-key"

# Train on SearchQA
python scripts/train.py \
    --config configs/searchqa/default.yaml \
    --split_dir /path/to/data \
    --optimizer_model gpt-5.5 \
    --target_model gpt-5.5

# Deploy the result
# → outputs/*/best_skill.md is ready to use!
```

### Pattern 2: Build from Source (Development)

```bash
# Clone official repository
git clone https://github.com/microsoft/SkillOpt.git
cd SkillOpt
pip install -e .

# Customize and extend
# → See docs/guide/new-backend.md
# → See docs/guide/new-benchmark.md
```

### Pattern 3: Custom Implementation (Research)

```python
# Use our reference implementation
from skillopt_implementation import SkillOptimizer

# Build your own optimizer
optimizer = SkillOptimizer(
    optimizer_model=your_optimizer_llm,
    target_model=your_target_llm,
    # ... configure hyperparameters
)

# Run training loop
best_skill = optimizer.optimize(train_data, val_data)
```

## ⚡ Key Insights (From All Sources)

### From the Paper

1. **Text-space optimization works:** Treating skills as "external trainable state" achieves fine-tuning-level gains without touching model weights.

2. **Validation gating is critical:** Only accept edits that improve held-out performance (strict improvement: score_new > score_current).

3. **Edit economy:** 1-4 accepted edits produce massive gains (+10 to +40 points). Most proposed edits are rejected.

4. **Transferability:** Skills work across models, harnesses, and benchmarks.

### From the Repository

5. **Production-ready:** The official package includes auto-resume, multi-backend support, and pre-trained reference skills.

6. **Slow-update mode matters:** Current default (force-accept) differs from paper protocol (gated). Use `slow_update_gate_with_selection: true` to reproduce paper results.

7. **Gate metric selection:** Hard gate (exact-match) is default, but soft/mixed gates help with small validation splits.

8. **Extensibility first:** The codebase is designed for adding custom backends and benchmarks.

### From Our Implementation

9. **Core components:** The 7 components (rollout, reflection, edits, budget, gate, buffer, slow update) are all essential. Ablations show removing any one degrades performance.

10. **Hyperparameter stability:** Default settings (4 epochs, batch 40, minibatch 8, lr 4) work well across benchmarks. You rarely need to tune.

## 🚨 Common Pitfalls & Solutions

| Pitfall | Symptom | Solution |
|---------|---------|----------|
| **Missing AZURE_OPENAI_ENDPOINT** | All LLM calls fail | Set this variable even for standard OpenAI |
| **Hard gate rejects everything** | No edits accepted | Switch to `gate_metric: soft` for small val splits |
| **Paper results don't match** | Lower gains than reported | Use `slow_update_gate_with_selection: true` |
| **Training crashes mid-run** | Loses progress, restarts from scratch | Check `runtime_state.json` exists; enable auto-resume |
